# 第 10 章：协调器模式

来源：[协调器模式 - 多智能体编排](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%B8%89%E9%83%A8%E5%88%86-%E9%AB%98%E7%BA%A7%E6%A8%A1%E5%BC%8F%E7%AF%87/10-%E5%8D%8F%E8%B0%83%E5%99%A8%E6%A8%A1%E5%BC%8F-%E5%A4%9A%E6%99%BA%E8%83%BD%E4%BD%93%E7%BC%96%E6%8E%92.md)

## 本章核心观点

本章把 Coordinator 模式定义为中心化多智能体编排：协调者管理 worker 生命周期、任务分配、消息传递和恢复，但不亲自执行工作。它与 Fork 的区别是：Fork 偏轻量平行分发，Coordinator 偏任务管理和企业级编排。

本章特别强调双重门控、模式恢复一致性、协调者工具集受限，以及 scratchpad 协作空间。

## 对照 Electrolyte 现有文档

已覆盖：

- Phase 5 已有 Long-Running Coding Agent。
- Phase 6 有 Automation and Extensibility。
- 架构文档有 Scheduler / Automation、State Layer、AgentRun。

部分覆盖：

- 现有 Phase 5 没有明确 coordinator 与 worker 的边界。
- 还没有“只编排不执行”的约束。
- 任务间 scratchpad、worker message、stop worker 等协议还未设计。

## 值得参考

值得参考的是 coordinator 的负面边界：它不应拿到所有执行工具，也不应亲自改文件。它应该拥有任务创建、停止、消息、状态读取和总结能力。

这能避免多智能体系统退化成一个全能 agent 加一堆不透明后台任务。

## 暂不采纳

不建议在近期实现 Coordinator。Electrolyte 当前还没有基础 long-running run、checkpoint、resume、worker cancellation，这些都是 coordinator 的前置条件。

## 后续建议

Phase 5 可以增加一条设计约束：如果引入 coordinator，必须先实现 worker ownership、write boundary、scratchpad storage、stop protocol 和 worker event projection。
