# 第 08 章：钩子系统

来源：[钩子系统 - Agent 的生命周期扩展点](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%BA%8C%E9%83%A8%E5%88%86-%E6%A0%B8%E5%BF%83%E7%B3%BB%E7%BB%9F%E7%AF%87/08-%E9%92%A9%E5%AD%90%E7%B3%BB%E7%BB%9F-Agent%E7%9A%84%E7%94%9F%E5%91%BD%E5%91%A8%E6%9C%9F%E6%89%A9%E5%B1%95%E7%82%B9.md)

## 本章核心观点

本章把 hook 视为生命周期扩展点，而不是随便运行的脚本。Hook 可以观察、修改输入、追加上下文、阻断操作，但必须有事件类型、输入输出 schema、优先级、安全门禁和执行结果语义。

书中区分 command、prompt、agent、HTTP、function 多种 hook，并强调 managed hooks、workspace trust 和配置来源安全。

## 对照 Electrolyte 现有文档

已覆盖：

- [AI Agent Runtime 设计](../../architecture/agent-runtime-design.md) 在 AgentDefinition 中有 LifecycleHooks。
- 插件和扩展章节提到扩展加载原则。
- 安全和 ToolRuntime 文档已经提供审批边界。

部分覆盖：

- Hook 在 Electrolyte 中还没有被详细设计为协议。
- Hook 与 permission pipeline 的关系尚未明确。
- Hook 的安全来源、可禁用能力面、输入输出 JSON schema 暂未展开。

## 值得参考

最值得参考的是 hook 不应直接拥有系统权限。Hook 可以参与决策或提供上下文，但最终执行仍应回到 ToolRuntime 和 Permission Pipeline。

如果 Electrolyte 后续实现 hooks，应从最少事件开始：SessionStart、PreToolUse、PostToolUse、Stop。不要一开始实现大量生命周期点。

## 暂不采纳

不建议早期支持 prompt hook 或 agent hook。它们会引入二次模型调用和复杂权限语义。Phase 0-2 最多考虑 command/function 级内部 hook。

## 后续建议

Hooks 应放到 Phase 6 或插件系统成熟后；若 Phase 3 skills 需要 hook，也只提供受限 pre/post lifecycle，并受 policy 锁定。
