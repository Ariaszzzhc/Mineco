# 第 04 章：权限管线

来源：[权限管线 - Agent 的护栏](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%B8%80%E9%83%A8%E5%88%86-%E5%9F%BA%E7%A1%80%E7%AF%87/04-%E6%9D%83%E9%99%90%E7%AE%A1%E7%BA%BF-Agent%E7%9A%84%E6%8A%A4%E6%A0%8F.md)

## 本章核心观点

本章把权限视为多阶段短路管线：输入验证、规则匹配、工具上下文评估、交互式提示。它的关键不是“是否询问用户”，而是权限决策必须有来源、优先级、上下文和不可覆盖的 deny。

书中还强调权限上下文不可变、用户/hook/classifier 多来源决策、ResolveOnce 竞争解决，以及 Bash 命令的精细规则匹配。

## 值得参考

最值得参考的是权限上下文作为不可变快照。工具并发执行时，权限检查必须基于本次请求创建时的稳定上下文，用户新批准的规则不应 retroactively 改变已在检查中的请求。

另一个重点是 ResolveOnce。未来如果 Mineco 同时支持用户 UI、policy hook、自动分类器审批，必须保证同一审批请求只有一个最终决定。

## 暂不采纳

不建议在 Phase 0 引入 classifier 自动审批。Phase 0 应只做明确规则和用户确认，减少不可解释授权。

## 后续建议

建议在权限模型中补 `decisionSource`、`decisionScope`、`expiresAt` 或等价字段，并明确 deny 优先级。
