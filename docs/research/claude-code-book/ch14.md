# 第 14 章：Plan 模式与结构化工作流

来源：[Plan 模式与结构化工作流](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E5%9B%9B%E9%83%A8%E5%88%86-%E5%B7%A5%E7%A8%8B%E5%AE%9E%E8%B7%B5%E7%AF%87/14-Plan%E6%A8%A1%E5%BC%8F%E4%B8%8E%E7%BB%93%E6%9E%84%E5%8C%96%E5%B7%A5%E4%BD%9C%E6%B5%81.md)

## 本章核心观点

本章把 Plan mode 定义为“先只读探索，再审批执行”的结构化工作流。它不是提示词风格，而是权限模式切换：规划阶段禁止写入和副作用，退出时保存计划并请求批准，执行后可由独立验证者对照计划检查。

书中还讨论原权限模式恢复、计划文件持久化、恢复策略和 team lead 审批路径。

## 对照 Mineco 现有文档

已覆盖：

- 开发流程文档要求设计先行和任务拆分。
- Phase 1/5 有 reviewability、resume、long-running agent。
- 架构文档有 Human-in-the-loop 和 checkpoint。

历史判断：

- 当时 Mineco 路线图还没有把 Plan mode 作为明确 runtime capability。
- 当时 approval 主要围绕 tool risk，而不是“计划批准”。
- 计划文件、计划恢复、计划验证者尚未进入数据模型。

当前状态：该判断已被 Phase 1 gate 采纳。最小 Plan mode 已明确为 Phase 1 必做 runtime permission mode：read-only permission mode、`plan.submit`、plan artifact、批准后恢复原 permission mode。复杂 plan recovery、独立 verifier 和 subagent/team lead 审批流保留到 Phase 5+。

## 值得参考

这是整本书对 Mineco 最值得补的点之一。Plan mode 可以显著降低 coding agent 的方向性错误，而且它和权限、安全、reviewability 都自然结合。

它不进入 Phase 0；最小版本已归入 Phase 1。Phase 5 只扩展复杂 plan recovery、独立 verifier、subagent/team lead 审批流和长任务中的计划对照。

## 暂不采纳

不建议把 Plan mode 只做成 slash command prompt。那样会失去最关键的安全边界。若实现，就应作为 permission mode。

## 后续建议

已在整体路线图中为 Plan mode 定位：最小版本属于 Phase 1，包括只读权限模式、计划 artifact、用户批准和轻量执行后对照检查；复杂恢复和独立验证属于 Phase 5+。
