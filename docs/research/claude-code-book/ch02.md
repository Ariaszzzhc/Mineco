# 第 02 章：对话循环

来源：[对话循环 - Agent 的心跳](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%B8%80%E9%83%A8%E5%88%86-%E5%9F%BA%E7%A1%80%E7%AF%87/02-%E5%AF%B9%E8%AF%9D%E5%BE%AA%E7%8E%AF-Agent%E7%9A%84%E5%BF%83%E8%B7%B3.md)

## 本章核心观点

本章把 agent loop 视为可暂停、可恢复、可取消的事件生产者。异步生成器的价值不是语法，而是它把模型流、工具进度、消息、墓碑消息、工具摘要和终止原因统一成一个有序事件流。

本章还强调三件事：一轮 turn 的生命周期、依赖注入对测试的价值、细粒度 terminal reason 对可观测性的价值。

## 对照 Mineco 现有文档

已覆盖：

- [Agent Runtime 协议](../../protocol/agent-runtime-protocol.md) 已经定义 `runStep(input): AsyncIterable<RunEvent>`。
- [Phase 0 本地 Agent 执行架构](../../roadmap/phase-0-local-agent-execution.md) 已经把 RuntimeEvent stream 作为核心接口。
- [AI Agent Runtime 设计](../../architecture/agent-runtime-design.md) 有 Agent Loop、状态机、中断和用户插话章节。

部分覆盖：

- Mineco 已有事件模型，但 terminal reason 还可以更细。协议当前有 Error Model 和 Run Events，但对 completed、max_turns、user_aborted、tool_denied、context_blocked、compaction_failed 等终止路径没有形成专门枚举。
- 依赖注入在文档中作为测试需求存在，但还没有明确列出 AgentLoopDeps 的最小依赖边界。

## 值得参考

最值得参考的是“yield 过程、return 结论”的结构。Mineco 的 runtime 事件应该表达过程，session/run status 应表达结论，二者不要混在单个字符串或单个 response object 中。

第二个参考点是“状态读写分界”：每轮开始读取稳定快照，每轮结束整体写入新状态。这个原则对 resume、replay 和并发工具执行都重要。

## 暂不采纳

不需要照搬 Claude Code 的具体消息类型名称，例如 TombstoneMessage。Mineco 可以在协议中保留等价的 invalidation 或 replacement event，等 UI 有真实需求时再细化。

## 后续建议

建议后续在 [Agent Runtime 协议](../../protocol/agent-runtime-protocol.md) 增加 TerminalReason 枚举，并在 Phase 0 中要求 MockProvider 测试覆盖至少三种终止路径。
