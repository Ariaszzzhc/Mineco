# 第 03 章：工具系统

来源：[工具系统 - Agent 的双手](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%B8%80%E9%83%A8%E5%88%86-%E5%9F%BA%E7%A1%80%E7%AF%87/03-%E5%B7%A5%E5%85%B7%E7%B3%BB%E7%BB%9F-Agent%E7%9A%84%E5%8F%8C%E6%89%8B.md)

## 本章核心观点

本章把 tool 设计成完整协议，而不是普通函数。一个工具至少包含名称、schema、权限模型、执行逻辑和 UI/进度表达。`buildTool` 一类工厂的价值在于填充安全默认值，使工具默认 fail-closed。

本章还强调工具注册、动态发现、并发分区和流式执行。只读工具可并行，有副作用工具应独占或保守串行。

## 对照 Electrolyte 现有文档

已覆盖：

- [AI Agent Runtime 设计](../../architecture/agent-runtime-design.md) 已有 Tool / MCP / Connector 统一模型和 Built-in Tools。
- [Agent Runtime 协议](../../protocol/agent-runtime-protocol.md) 已有 Tool Specification、ToolPolicy、ToolPermissions、Tool Runtime。
- Phase 0 已列出 `file.list`、`file.search`、`file.read`、`file.write`、`shell.run`。

部分覆盖：

- 当前 ToolSpec 已经有 schema 和 policy，但工具的进度事件、输出 artifact 策略、并发安全属性和 UI 生命周期没有像本章一样形成完整五要素。
- Phase 0 对工具并发的范围较轻，后续可能需要明确 read-only tools 可并行、write/shell 串行。

## 值得参考

最值得参考的是工具定义的完整度。Electrolyte 不必在 Phase 0 实现复杂 UI render 方法，但 ToolDefinition 应该尽早记录：

- 是否只读。
- 是否 destructive。
- 是否 concurrency-safe。
- 结果最大内联大小和 artifact 策略。
- progress event 的结构。

这些字段越晚补，越容易破坏 provider、tool runtime 和 UI 的边界。

## 暂不采纳

不建议在 Phase 0 过早实现延迟工具发现。Electrolyte Phase 0 只有少量内置工具，工具 schema 膨胀不是首要问题。该机制应放到 Phase 2 MCP 工具数量变多后再做。

## 后续建议

建议后续补充 ToolDefinition 的“执行元数据”字段，并在 Phase 0 的 tool policy tests 中加入并发安全分类测试。
