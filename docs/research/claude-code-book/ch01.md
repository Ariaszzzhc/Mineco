# 第 01 章：智能体编程的新范式

来源：[智能体编程的新范式](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%B8%80%E9%83%A8%E5%88%86-%E5%9F%BA%E7%A1%80%E7%AF%87/01-%E6%99%BA%E8%83%BD%E4%BD%93%E7%BC%96%E7%A8%8B%E7%9A%84%E6%96%B0%E8%8C%83%E5%BC%8F.md)

## 本章核心观点

本章把 Agent Harness 定义为围绕 LLM 构建的运行时框架，而不是模型 API 的轻量封装。重点不是“如何调用模型”，而是模型如何通过工具、权限、状态、上下文、错误恢复和可扩展边界成为可靠的任务执行者。

书中强调的五类工程问题是：工具注册与发现、参数校验、权限管控、错误恢复、状态一致性、并发调度。它们共同说明 agent runtime 的复杂度来自系统工程，而不是 prompt。

## 对照 Electrolyte 现有文档

已覆盖：

- [AI Agent Runtime 设计](../../architecture/agent-runtime-design.md) 的目标已经明确 Electrolyte 不是 chat API wrapper。
- 分层架构中已经把 Agent Core、Provider Layer、Tool Execution Layer、State Layer 分开。
- [Agent Runtime 协议](../../protocol/agent-runtime-protocol.md) 已写明 Provider Adapter 不执行工具，Tool Runtime 不决定模型行为。

部分覆盖：

- 现有文档覆盖了“原则”，但还缺少一个面向评审的判断：任何新增能力是否真的服务于 agent execution，而不是普通 chat 封装。
- 技术栈选择已有 Node.js 25-first 方向，但还没有像本章那样明确讨论“终端 UI 不应只是 console 拼接”的产品体验原则。

## 值得参考

最值得参考的是“简单封装 vs Harness”的判断框架。Electrolyte 目前的长期设计方向是正确的，但 Phase 0 容易因为追求最小切片而退化成简单循环。Phase 0 文档中应持续保留 runtime event、session store、tool runtime 和 approval，而不能为了快跑把这些压到后面。

## 暂不采纳

书中关于 Bun、React/Ink、Zod v4 的技术栈结论不直接采纳。Electrolyte 当前基线是 Node.js 25、pnpm workspace、Vitest、Kysely、Effect 和 Zod；需要吸收的是“运行时选择服务于启动速度、类型安全和可测试性”的原则，而不是具体照搬书中选型。

## 后续建议

无需立即修改主设计文档。可在进入实现前，把 Phase 0 的 review gate 增加一条：新增代码不能绕过 RuntimeEvent、ToolRuntime 和 Store 边界直接调用 provider 和工具。
