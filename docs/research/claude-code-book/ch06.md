# 第 06 章：记忆系统

来源：[记忆系统 - Agent 的长期记忆](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%BA%8C%E9%83%A8%E5%88%86-%E6%A0%B8%E5%BF%83%E7%B3%BB%E7%BB%9F%E7%AF%87/06-%E8%AE%B0%E5%BF%86%E7%B3%BB%E7%BB%9F-Agent%E7%9A%84%E9%95%BF%E6%9C%9F%E8%AE%B0%E5%BF%86.md)

## 本章核心观点

本章最重要的观点是：memory 只保存不可从当前项目状态推导的信息。代码结构、依赖版本、路由列表、Git 历史都可以重新读取，不应进入长期记忆。

书中推荐闭合类型系统：user、feedback、project、reference。每类记忆都要说明何时保存、如何使用。相对日期必须转成绝对日期。

## 对照 Electrolyte 现有文档

已覆盖：

- [AI Agent Runtime 设计](../../architecture/agent-runtime-design.md) 已有 Memory 类型和 MemoryPolicy。
- 长期记忆被放在 Phase 0 非目标之外，避免第一阶段过早复杂化。

部分覆盖：

- 现有 Memory 类型还可以更明确地区分“可推导”和“不可推导”。
- 现有文档没有强调 memory 不是 docs，也不是代码索引。
- 相对日期转绝对日期这一点值得加入流程规则。

## 值得参考

最值得参考的是保存准则，而不是实现机制。Electrolyte 后续做 memory 时，应先定义“什么不能保存”，否则 memory 会变成噪音源，污染上下文。

建议闭合类型可以直接参考，但命名不必完全一致。

## 暂不采纳

不建议在 Phase 0 或 Phase 1 实现完整长期记忆。现有 roadmap 把长期记忆延后是合理的。早期更需要 event log、artifact 和 replay。

## 后续建议

后续进入 memory 设计时，在文档里增加“不保存可由工具重新获取的信息”作为硬规则，并要求 memory item 写明 Why 和 How to apply。
