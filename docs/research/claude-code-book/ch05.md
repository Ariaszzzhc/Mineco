# 第 05 章：设置与配置

来源：[设置与配置 - Agent 的基因](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%BA%8C%E9%83%A8%E5%88%86-%E6%A0%B8%E5%BF%83%E7%B3%BB%E7%BB%9F%E7%AF%87/05-%E8%AE%BE%E7%BD%AE%E4%B8%8E%E9%85%8D%E7%BD%AE-Agent%E7%9A%84%E5%9F%BA%E5%9B%A0.md)

## 本章核心观点

本章重点是配置来源、合并规则和信任边界。配置不是单一文件，而是由插件、用户、项目、本地、CLI、企业策略等来源叠加。不同字段有不同合并策略；安全敏感字段要区别可信来源和不可信项目配置。

书中对 project settings 的供应链攻击风险分析尤其重要：克隆来的仓库配置不能自动扩大执行权限或启用危险 hook。

## 对照 Mineco 现有文档

已覆盖：

- [AI Agent Runtime 设计](../../architecture/agent-runtime-design.md) 已有配置格式章节。
- AgentDefinition 已支持系统内置、工作区和用户自定义来源。
- Phase 0 关注 workspace boundary 和基础配置。

部分覆盖：

- 现有配置文档还没有形成清晰的 config source priority 和 merge semantics。
- 对 project/workspace 配置的信任边界还不够明确，尤其是权限、hooks、MCP servers、plugins、skills 这些可扩大能力面的配置。
- policy/enterprise 配置在长期设计里有影子，但还不是一等来源模型。

## 值得参考

最值得参考的是“配置来源即信任来源”。Mineco 不能只定义一个 `settings.json`。至少应在长期设计中区分 user config、workspace shared config、workspace local config、CLI override、policy config。

安全敏感项要有明确规则：workspace shared config 可以提供项目约定，但不能开启危险自动批准、外部连接或未审核 hook。

## 暂不采纳

不需要照搬 Claude Code 的具体文件路径和企业 MDM 机制。Mineco 当前阶段只需要先定义抽象来源和信任规则。
