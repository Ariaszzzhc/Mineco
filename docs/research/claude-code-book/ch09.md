# 第 09 章：子智能体与 Fork 模式

来源：[子智能体与 Fork 模式](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E4%B8%89%E9%83%A8%E5%88%86-%E9%AB%98%E7%BA%A7%E6%A8%A1%E5%BC%8F%E7%AF%87/09-%E5%AD%90%E6%99%BA%E8%83%BD%E4%BD%93%E4%B8%8EFork%E6%A8%A1%E5%BC%8F.md)

## 本章核心观点

本章讨论 subagent 的角色定义、来源、生命周期和 Fork 模式。重点不是“多开几个 agent”，而是每个 agent 要有明确能力边界：只读探索、规划、通用执行、对抗性验证等角色不同，工具权限也不同。

Fork 模式的核心是继承父上下文快照并尽量保持 prompt cache 命中，同时避免共享可变父状态。

## 对照 Mineco 现有文档

## 值得参考

最值得参考的是角色分离：explorer 只读，planner 只读，worker 可写，verifier 对抗性只读。这对 Mineco 未来长任务非常有价值。

即使不实现 subagent，也可以先在工作流里吸收这种分工：读代码、计划、修改、验证不要都由同一段逻辑混在一起完成。

## 暂不采纳

不建议 Phase 0-2 实现 Fork 模式。它依赖 prompt caching、稳定 provider state、session fork、任务生命周期管理，过早实现会增加复杂度。

## 后续建议

Phase 5 文档可以提前写明：subagent 必须声明工具权限、写入范围、是否后台运行、最大轮次和取消传播。
