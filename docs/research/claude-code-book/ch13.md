# 第 13 章：流式架构与性能优化

来源：[流式架构与性能优化](https://github.com/lintsinghua/claude-code-book/blob/main/%E7%AC%AC%E5%9B%9B%E9%83%A8%E5%88%86-%E5%B7%A5%E7%A8%8B%E5%AE%9E%E8%B7%B5%E7%AF%87/13-%E6%B5%81%E5%BC%8F%E6%9E%B6%E6%9E%84%E4%B8%8E%E6%80%A7%E8%83%BD%E4%BC%98%E5%8C%96.md)

## 本章核心观点

本章强调流式架构不是表层优化，而是 agent harness 的整体架构选择。模型 token、tool call block、tool progress 和 usage 都应在流中增量处理。

StreamingToolExecutor 的关键是：工具可尽早启动，但并发要受安全属性限制；进度可即时产出，最终结果要保持模型可理解的顺序。

## 对照 Mineco 现有文档

已覆盖：

- [Agent Runtime 协议](../../protocol/agent-runtime-protocol.md) 已定义 Streaming Agent Events。
- Phase 0 要求 assistant output、tool call 和 tool result 可见；streaming 是 provider 可选能力。
- 架构文档有 Product Surface Runtime Boundary。

部分覆盖：

- 现有文档还没有明确工具流式执行器和结果顺序策略。
- Usage event 有协议定义，但成本/延迟指标与 runtime observability 的关系还可以加强。
- 启动性能和延迟预算尚未在开发 gate 中体现。

## 值得参考

最值得参考的是“流到即执行，但保守并发”。Mineco 应允许 Read/Search 类工具并发，但 shell/write/edit 默认串行。即使结果并行完成，也要按 tool call 顺序写入 transcript，避免模型观察错位。

## 暂不采纳

不建议 Phase 0 做增量 JSON 解析和工具预启动。先等 provider event 和 tool call item 稳定，再优化延迟。

## 后续建议

Phase 1 或 Phase 4 后可补 StreamingToolExecutor 设计，明确 queued、executing、completed、yielded 状态，以及 progress 与 final result 的输出顺序。
